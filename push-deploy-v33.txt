# What changed in v33:
#   FIX "Invalid MPID Format" crash on Materials Project queries.
#   Root cause: the live MP API now returns alias material_ids like
#   "mp-aaaaaaft" (letter suffix). They match NEITHER emmet-core's MPID regex
#   NOR the ULID regex, so EVERY emmet-core version rejects them and the whole
#   query dies. Bumping/downgrading emmet-core/mp-api/pymatgen does NOT fix it.
#   Fix lives in src/tool/tool_mp.py: MPRester(use_document_model=False) ->
#   raw dicts, bypassing validation, plus an _as_structure() compatibility
#   shim (nested structures decode to Structure on mp-api 0.45.x but stay dicts
#   on 0.46.x — that is the real "prod != sandbox" trap).
#   Verified end-to-end on:
#     "Show me the band structure of silicon, diamond cubic, using vc-relax with LDA"
#     -> material_id mp-aaaaaaft, Fd-3m (227), a=3.849 A, NO crash,
#     identical result on emmet-core 0.85.1 AND 0.86.4.
#
# Image tags are IMMUTABLE: code changes REQUIRE a new tag. Both the API
# (tritondft) and the worker (tritondft-worker) deployments use the SAME image
# — bump BOTH. k8s/deployment.yaml and k8s/worker.yaml already pin :v33.
# =============================================================================

# =============================================================================
# CONTINGENCY DURING ROLLOUT: a NEW v33 pod can't resolve 'postgres'
# (folded in from troubleshoot-dns-pod.txt — this is a SEPARATE issue)
# =============================================================================
# Symptom: ONE (or a few) pod loops on
#   claim loop error: failed to resolve host 'postgres'
#   reap_stale  error: failed to resolve host 'postgres'
# while the other workers poll normally.
#
# This is NOT an app bug and NOT related to the v33 code change. It is a
# per-node in-cluster DNS failure (CoreDNS / node-local-dns on that node) on
# NRP. A rollout reschedules pods, so a fresh pod can land on a bad node and
# show this. Fix = get that pod off the bad node (reschedule), NOT a redeploy.
#
# Every command below AUTO-DETECTS the failing pod(s). Copy-paste as-is — there
# is nothing to fill in by hand.

# ---- C1. Auto-detect which worker pods are wedged on DNS --------------------
# (scans each worker's recent logs for the postgres resolve error)
BAD=$(for p in $(kubectl -n datahub-llm get pods -l k8s-app=tritondft-worker -o name); do \
  kubectl -n datahub-llm logs --tail=20 "$p" 2>/dev/null \
    | grep -q "failed to resolve host 'postgres'" && echo "${p#pod/}"; \
done); echo "FAILING PODS: ${BAD:-(none)}"

# ---- C2. Delete the wedged pod(s) (Deployment recreates them elsewhere) -----
[ -n "$BAD" ] && kubectl -n datahub-llm delete pod $BAD || echo "nothing to delete"

# ---- C3. Watch the replacement(s) come back clean --------------------------
kubectl -n datahub-llm logs -l k8s-app=tritondft-worker --tail=50 -f --max-log-requests 8
# Expect: new pod prints "agent loaded, polling for jobs" and NO
# "failed to resolve host 'postgres'". (While a new pod is ContainerCreating,
# `logs -f` may end with "context deadline exceeded" / "closed pipe" — that is
# the log-follow giving up on a not-yet-started container, not an app error.
# Just wait and re-tail.)

# ---- C4. If replacements KEEP failing: cordon the bad node(s), then delete --
# Re-detects the currently-failing pods, cordons whatever node each sits on so
# the scheduler avoids it, then deletes them to force a reschedule elsewhere.
BAD=$(for p in $(kubectl -n datahub-llm get pods -l k8s-app=tritondft-worker -o name); do \
  kubectl -n datahub-llm logs --tail=20 "$p" 2>/dev/null \
    | grep -q "failed to resolve host 'postgres'" && echo "${p#pod/}"; \
done); \
NODES=$(for p in $BAD; do \
  kubectl -n datahub-llm get pod "$p" -o jsonpath='{.spec.nodeName}{"\n"}'; \
done | sort -u); \
for n in $NODES; do echo "cordoning $n"; kubectl cordon "$n"; done; \
[ -n "$BAD" ] && kubectl -n datahub-llm delete pod $BAD; \
echo "=== SAVE THIS to uncordon later ==="; \
echo "kubectl uncordon $NODES"
# ^ The last line prints a ready-to-run uncordon command listing ONLY the nodes
#   you just cordoned. Copy it somewhere and run it once NRP fixes those nodes.
#   (Do NOT blanket-uncordon — other tenants may have cordoned other nodes.)

# ---- C5. Sanity-check cluster DNS / postgres are otherwise healthy ----------
kubectl -n datahub-llm get svc postgres
kubectl -n datahub-llm get endpoints postgres                      # must have a ready endpoint
kubectl -n datahub-llm get pods -l app=postgres -o wide
# If ALL workers fail to resolve 'postgres' (not just one) OR endpoints is
# empty, it's cluster-wide DNS or the postgres pod is down — not a single bad
# node; investigate postgres itself.
